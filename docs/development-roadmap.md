# Native OS Development Roadmap (Detailed)

最終更新: 2026-03-04

## 1. 目標

- 最終目標: 実機で継続運用できる本格OS
- 中期目標: Linuxアプリ互換に向けた土台を段階的に実装
- 直近目標: 現在のシェル/入力/描画基盤を壊さずに、ユーザーモード実行へ進む

## 2. 基本方針

- Linux完全互換を最初から狙わない
- まずは「自作カーネルとして堅い基盤」を作る
- その上に POSIX サブセットを実装し、最後に Linux syscall 互換を広げる
- 各段階で「動作確認できる成果物」を必ず作る

## 3. フェーズ計画

### Phase 0: 基盤整備（進行中）

目的:
- カーネルの保守性向上
- 巨大ファイル/巨大関数の分割

現状:
- `shell` は `cmd_dispatch/cmd_core/cmd_fs/cmd_xhci` に分割済み
- 各コマンド群は別オブジェクトとしてコンパイル/リンク済み
- イベントループ内の一部ロジック分割済み（xHCI idle処理、マウス処理、拡張キー処理、通常ショートカット処理）

完了条件:
- `kernel.cpp` のイベントループ責務をさらに分割
- 回帰チェック手順が README で運用される

---

### Phase 1: ユーザーモード実行 (ring3) と syscall 入口

目的:
- カーネル空間とユーザー空間を分離
- 将来のアプリ実行の最小骨格を作る

実装タスク:
1. TSS/GDT 拡張（ring3 セグメント）
2. `syscall/sysret` または `int 0x80` 入口の実装
3. ユーザースタックの安全な初期化
4. 最小ユーザープログラム（`hello`）のロードと実行
5. カーネル戻りとエラー処理

完了条件:
- ring3 の `hello` が起動し、`write` 相当の syscall で文字を表示

---

### Phase 2: プロセス・仮想メモリ管理

目的:
- 1プロセス実行から複数プロセスへ拡張

実装タスク:
1. PCB（プロセス管理構造体）
2. スケジューリング最小実装（ラウンドロビン）
3. `fork/exec/exit/wait` の最小動作
4. アドレス空間分離（ユーザー毎のページテーブル）
5. `brk/mmap` の最小版

完了条件:
- 2つ以上のユーザープログラムを順次実行可能

---

### Phase 3: 仮想FSから実ディスクFSへ

目的:
- 再起動で消えないファイルシステム

実装タスク:
1. ブロックI/O抽象層（read/write block）
2. ストレージドライバ（AHCI または NVMe のどちらか優先）
3. FAT32 読み書きの最小実装（最初は single-partition 前提）
4. 既存 shell FS コマンドのバックエンド差し替え
5. エラー回復・整合性チェック

完了条件:
- `write` したファイルが再起動後も読める

---

### Phase 4: POSIX サブセット

目的:
- 既存Cアプリ移植のための API 面を整える

対象API（最小）:
- `open/read/write/close/lseek`
- `stat/fstat`
- `dup/dup2`
- `pipe`
- `getpid`

実装タスク:
1. FDテーブル
2. vnode/ファイルオブジェクト層
3. パイプバッファ
4. `errno` と失敗コード統一

完了条件:
- `cat`, `echo`, `sh` 相当のユーザープログラムが動く

---

### Phase 5: Cランタイム + ユーザーランド拡張

目的:
- 既存ソフトを段階的に実行可能にする

実装タスク:
1. `newlib` か `musl` のどちらかを先に接続
2. 最小 libc/syscall ブリッジ
3. BusyBox の一部コマンド移植
4. 起動時にユーザー空間 `init` 実行

完了条件:
- ユーザー空間シェルで複数コマンドが安定動作

---

### Phase 6: Linux 互換層（部分）

目的:
- 既存 Linux CLI アプリの実行対象を増やす

実装タスク:
1. syscall 番号マッピング
2. 不足 syscall の段階実装
3. `poll/select/epoll` の最小対応
4. `signal` の最小対応

完了条件:
- ターゲットCLIアプリが実行できる（対象を固定して達成）

## 4. 品質管理

各機能追加時の固定チェック:
1. `.\scripts\build.ps1` 成功
2. 起動後プロンプト確認
3. 主要 shell コマンド確認（READMEチェックリスト）
4. 関連機能の手動回帰確認
5. commit + push

## 5. ブランチ運用ルール

- `main`: 常に起動可能状態を維持
- 大きな実装は小分けコミット（1コミット1責務）
- 壊れた状態を push しない

## 6. 当面（次の2週間）タスク

1. イベントループの継続分割（keyboard通常入力の関数分離）
2. `kernel/input` 配下へ入力処理ロジック移設
3. syscall 入口の最小実装設計（ABI, レジスタ規約, エラー規約）
4. ring3 `hello` 実行 PoC の着手

## 7. リスクと対策

- リスク: Linux互換を急ぎすぎてカーネル基盤が不安定化
- 対策: フェーズ順を固定し、Phase 2完了前に互換拡張へ進まない

- リスク: ドライバ実装の複雑化で停滞
- 対策: 先に「最小対応デバイス」を1つ決めて完了まで絞る

- リスク: 回帰バグ増加
- 対策: README手動テストチェックリストを毎回実施
